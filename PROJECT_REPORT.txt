==============================================================================
1. PROJECT OVERVIEW & ARCHITECTURE
==============================================================================
ATMOSCHAIN is a tripartite system architecture consisting of a Computer Vision 
pipeline, a thermodynamic simulation engine, and an automated ERC-20 token 
generation system. It aims to solve the problem of open-air landfill emissions 
(specifically methane) by theoretically redirecting Municipal Solid Waste (MSW) 
to high-temperature Plasma Gasification plants and rewarding participants via 
tokenized carbon credits.

The system is composed of three tightly coupled modules:
  MODULE 1: WasteVision AI (Machine Learning & Computer Vision)
  MODULE 2: PlasmaSim Optimizer (Thermodynamic & Physics Simulation)
  MODULE 3: CCTS SmartMarket (Blockchain Tokenization & Ledger)

The user flow is sequential: 
  A user scans waste -> Gets AI Analysis -> Simulates its destruction -> Mints Carbon Credits.

==============================================================================
2. MACHINE LEARNING: WASTEVISION AI & METHANE ENGINE
==============================================================================
The primary purpose of the Machine Learning integration is to identify waste at 
the source and calculate its real-world climate hazard (Methane Potential).

A. Image Classification (Computer Vision)
- Tool used: Google Gemini 2.0 Flash Vision API.
- Why: Real-time, highly accurate edge-case detection. It analyzes a base64 
  webcam feed and returns structured JSON describing the waste (plastic, 
  organic, metal, etc.).
- Location: `ml_models/wastevision/waste_detector.py`

B. The Methane Prediction Engine (The core Mathematical ML Model)
- Tool used: IPCC (Intergovernmental Panel on Climate Change) First-Order Decay (FOD) Model.
- How it works:
  1. Once the waste is classified (e.g., "Organic Apple Core"), the model maps 
     this to a specific Degradable Organic Carbon (DOC) fraction. 
     (e.g., Food Waste DOC = 0.15).
  2. It applies a Methane Generation Rate Constant (k) and the Fraction of 
     Degradable Organic Carbon that decomposes (DOC_f).
  3. It calculates exactly how many kilograms of CH4 (Methane) this item 
     would produce over its lifetime in an anaerobic landfill like Ghazipur.
  4. It converts CH4 to CO2-equivalent (CO2e) using the Global Warming Potential 
     (GWP) multiplier of 28.
- Location: `ml_models/methane_model/methane_prediction.py`
- Defense: This is not just an API wrapper; it is an intelligent mathematical 
  pipeline applying international baseline climate science to real-time imagery.

==============================================================================
3. SIMULATION MODELING: PLASMASIM OPTIMIZER
==============================================================================
Plasma Gasification is the process of using 5000°C plasma arcs to break down 
waste at the molecular level, creating clean Syngas instead of toxic ash/smoke.

A. The Thermodynamic Pipeline
- Rather than just producing random numbers, this module simulates the four stages 
  of a real industrial gasification plant:
  1. Plasma Reactor (Thermal Cracking at 5500°C)
  2. Cyclone Separator (Removing >99% of >10-micron particles/slag)
  3. Gas Scrubber (Neutralizing acidic gases like HCl or H2S)
  4. Fractionation Tower (Cryogenically separating H2, CO, CO2, and CH4)
- Location: `simulation/` directory.

B. Gas Yield & Energy Prediction Modeling
- Based on the input waste type (from WasteVision), it uses an empirical database 
  (`ml_models/plasmasim_models/gas_yield_predictor.py`) to determine the exact 
  volumetric yield of Syngas (Hydrogen vs. Carbon Monoxide).
- It calculates the theoretical thermal and electrical energy output (kWh) 
  accounting for realistic turbine efficiencies (35%).
- Defense: The simulation is rooted in actual thermodynamic yield coefficients, 
  allowing the UI to visually and mathematically represent the destruction of waste.

==============================================================================
4. BLOCKCHAIN INTEGRATION: CCTS SMARTMARKET
==============================================================================
CCTS (Carbon Credit Trading System) decentralizes the issuance and tracking of 
carbon offsets, preventing double-counting and central authority manipulation.

A. The Smart Contract
- Standard: ERC-20 Token (Fungible Token).
- Name/Symbol: CarbonCreditToken (CCT).
- Location: `blockchain/smart_contracts/carbon_credit_token.sol`.
- Functionality: 
  - `mint()`: Allows the protocol (onlyOwner) to generate entirely new tokens 
    based off verified metric tonnes of CO2e avoided.
  - `retire()`: Allows a corporate buyer (e.g., an airline) to permanently "burn" 
    the token, claiming the environmental benefit for their regulatory compliance.
  - `CreditRecord`: Every minted batch maps to immutable metadata recorded on-chain, 
    stating exactly what caused the reduction (e.g., "WasteVision Detection - Plastic Bottle").

B. Integration in the Ecosystem
- Once WasteVision and PlasmaSim verify that X amount of Methane was diverted 
  from a landfill and converted into clean energy, the backend calculates the 
  equivalent Carbon Credits.
- The user clicks "Mint", which interactively sends a payload to the backend to
  "mint" those tokens into the simulated ledger (`backend/services/carbon_credit_engine.py`).
- The market dashboard lists available credits from both our project and seeded 
  external baseline projects (like Reforestation), allowing simulated industries 
  to buy and retire them.
- Defense: This provides a mathematically backed, transparent tracking mechanism 
  connecting physical waste destruction directly to financial liquidity instruments.

6. HOW TO RUN THE SYSTEM
==============================================================================
1. Ensure `GEMINI_API_KEY` is loaded into the `/.env` file.
2. Open Terminal 1:
   -> `cd d:\Projects\ATMOSCHAIN`
   -> `python -m uvicorn backend.api.app:app --reload --port 8000`
3. Open Terminal 2:
   -> `cd d:\Projects\ATMOSCHAIN\frontend`
   -> `npm run dev` (Runs on port 3000)
4. Visit `http://localhost:3000` and enjoy the interconnected experience.

A detailed breakdown of every source file, what it does, its inputs/outputs,

──────────────────────────────────────────────────────────────────────────────
MODULE: simulation/  —  PlasmaSim Gas Cleanup Pipeline
──────────────────────────────────────────────────────────────────────────────

  cyclone_separator_model.py
  ──────────────────────────
  ATMOSCHAIN — PlasmaSim Optimizer
  Models Stage 1 of gas cleanup: cyclone separator.
  Removes particulates, slag droplets, and heavy aerosols from hot syngas.

  Class   : CycloneSeparator
  Input   : raw_gas_pct (dict), input_mass_kg (float)
  Output  : clean_gas_pct, slag_kg, particulate_ppm, removal_efficiency_pct
  Physics : 99.2% removal efficiency for particles > 10 µm
  Conds   : 800 °C / 1.1 bar
  Notes   : Centrifugal force vitrifies and drops slag. Output feeds Stage 2.

  ─────────────────────────────────────────────────────────────────────────

  gas_scrubber_model.py
  ─────────────────────
  ATMOSCHAIN — PlasmaSim Optimizer
  Models Stage 2: wet gas scrubber.
  Removes acid gases (HCl, H2S), sulfur compounds, heavy metals, and dioxins.
  Uses NaOH or Ca(OH)2 scrubbing solution.

  Class   : GasScrubber
  Input   : input_gas_pct (dict — from cyclone output)
  Output  : clean_gas_pct, sulfur_removed_ppm, hcl_removed_ppm,
            post_scrubber_levels, eu_emission_standard compliance status
  Removes : SO2 (99.9%), HCl (99.5%), dioxins (99.97%), heavy metals (99%)
  Conds   : 400 °C / 1.05 bar
  Notes   : Checks dioxin level against EU WtE standard (0.1 ng/Nm3 limit).

  ─────────────────────────────────────────────────────────────────────────

  fractionation_tower_model.py
  ────────────────────────────
  ATMOSCHAIN — PlasmaSim Optimizer
  Models Stage 3: cryogenic fractionation / pressure swing adsorption (PSA).
  Separates syngas into individual product streams:
  H2, CO, CO2, CH4 — each with high purity for industrial use.

  Class   : FractionationTower
  Input   : input_gas_pct (dict — from scrubber output)
  Output  : fractionated_gas_pct, product_streams (per-gas purity, recovery %)
  Method  : Pressure Swing Adsorption (PSA) + cryogenic distillation
  Purity  : H2=99.9%, CO=98.5%, CO2=99.0%, CH4=97.0%
  Conds   : -180 °C / 3.0 bar (cryogenic)
  Notes   : H2 stream routed to fuel cells. CO for Fischer-Tropsch synthesis.

  ─────────────────────────────────────────────────────────────────────────

  gas_separation_pipeline.py
  ──────────────────────────
  ATMOSCHAIN — PlasmaSim Optimizer
  Orchestrates all gas separation stages:
    Cyclone → Scrubber → Fractionation Tower
  Provides per-stage removal efficiency and output purity data.

  Class   : GasSeparationPipeline
  Input   : raw_gas_pct (dict), input_mass_kg (float)
  Output  : stage_1_cyclone, stage_2_scrubber, stage_3_fractionation,
            final_gas_pct, pipeline_summary (key metrics dict)
  Depends : CycloneSeparator, GasScrubber, FractionationTower
  Run     : python simulation/gas_separation_pipeline.py

  ─────────────────────────────────────────────────────────────────────────

  plasma_reactor_sim.py
  ─────────────────────
  ATMOSCHAIN — PlasmaSim Optimizer
  Full plasma gasification reactor simulation. Models the state machine:
    WASTE_INPUT → PLASMA_ARC → THERMAL_CRACKING → CYCLONE_SEPARATION
    → GAS_SCRUBBING → FRACTIONATION → GAS_UTILIZATION

  Integrates gas_yield_predictor and energy_prediction to produce a complete
  per-stage simulation result for the frontend pipeline animation.

  Class   : PlasmaReactorSim
  Input   : waste_class (str), mass_kg (float)
  Output  : stages[] (7 per-stage dicts), gas_composition, gas_utilization,
            energy (kWh + revenue), slag (mass + use), summary (dashboard cards)
  Depends : GasYieldPredictor, EnergyPredictor
  Run     : python simulation/plasma_reactor_sim.py

MODULE: ml_models/methane_model/  —  Methane Prediction
──────────────────────────────────────────────────────────────────────────────

  methane_prediction.py
  ─────────────────────
  ATMOSCHAIN — WasteVision AI
  IPCC First-Order Decay (FOD) methane model for landfill gas estimation.
  Reference: IPCC 2006 Guidelines for National GHG Inventories, Vol. 5, Ch. 3

  IPCC Formula:
    CH4 (tonnes) = mass_tonnes × DOC × DOCf × F × MCF × (16/12)
    CO2e = CH4_tonnes × GWP100  (GWP100 = 28, IPCC AR6)

  Class   : MethanePredictor
  Input   : waste_class (str), mass_kg (float), doc_override (optional)
  Output  : ch4_kg, ch4_cubic_meters, co2e_kg, co2e_tonnes, carbon_credits,
            revenue_usd_min/mid/max, revenue_inr_mid, gwp_factor
  Bonus   : predict_landfill_daily() — aggregate for full landfill mix
            ghazipur_scenario()      — Ghazipur 2,800 t/day real scenario
  Run     : python ml_models/methane_model/methane_prediction.py

  ─────────────────────────────────────────────────────────────────────────

  train_methane_model.py
  ──────────────────────
  ATMOSCHAIN — Methane Model
  Trains a scikit-learn Random Forest Regressor on methane_historical.csv
  to predict CH4 yield from waste properties.

  Trains TWO RandomForestRegressor pipelines:
    ch4_predictor.pkl   — predicts ch4_yield_kg
    co2e_predictor.pkl  — predicts co2e_yield_kg

  Features: waste_type, mass_kg, doc_fraction, temperature_c,
            moisture_percent, mass_x_doc (engineered), log_mass (engineered)
  Notes   : Zero-yield materials (plastic/metal/glass) are handled by the
            physics model; this RF only trains on organic materials.
  Run     : python ml_models/methane_model/train_methane_model.py

8. CONTINUOUS DATA FLOW & ECOSYSTEM LIFECYCLE
==============================================================================
The true power of ATMOSCHAIN lies not in isolated models, but in the seamless 
continuous data flow across the entire architecture. Here is how folders and files
work together to create a cohesive lifecycle:

1. DATA INGESTION (Frontend -> WasteVision AI)
   - Flow: `frontend/components/digital-twin/WebcamDetector.tsx` captures live frames 
     and sends base64 streams via WebSockets to `backend/api/app.py`.
   - Function: Acts as the physical-to-digital bridge. Real-world physical objects 
     (waste) enter the digital system here.

2. CLASSIFICATION & YIELD PREDICTION (WasteVision -> Methane Engine)
   - Flow: `backend/api/wastevision_routes.py` routes the image to `ml_models/wastevision/waste_detector.py`.
     Once identified (e.g., Organic Waste), data flows to `ml_models/methane_model/methane_prediction.py`.
   - Function: Transforms raw pixels into actionable scientific metrics (Mass, DOC 
     fraction, CH4 emission potential). It quantifies the problem.

3. DIGITAL DESTRUCTION (Methane Engine -> PlasmaSim Optimizer)
   - Flow: The quantified waste payload is handed to `backend/api/plasmasim_routes.py`, 
     which triggers `simulation/plasma_reactor_sim.py`. 
   - Function: Instead of a physical landfill, the waste is virtually destroyed. 
     The simulation files (`cyclone_separator_model.py`, `gas_scrubber_model.py`, etc.) 
     process the payload step-by-step, transforming hazard metrics into clean 
     energy metrics (Syngas, Electricity) using `ml_models/plasmasim_models/`.

4. VALUE CREATION (PlasmaSim -> CCTS Blockchain)
   - Flow: The net positive impact (CO2e avoided) is pushed to `backend/services/carbon_credit_engine.py`, 
     which interfaces with `blockchain/smart_contracts/carbon_credit_token.sol`.
   - Function: Transforms the physical act of recycling/destruction into a fungible 
     financial asset (Carbon Credit Token). Data becomes currency.

5. FEEDBACK & MACRO-ANALYSIS (CCTS -> Analytics)
   - Flow: System-wide processing rates and credits are regularly aggregated by 
     `analytics/environmental_impact_model.py` and `revenue_projection.py`.
   - Function: Feeds the generated data into long-term projection models to adjust 
     plant ROI calculations and update city-wide heatmaps.

This pipeline ensures that every single module depends on the successful execution 
of the previous one. The continuous flow turns an image of a plastic bottle into 
a mathematically validated carbon credit within milliseconds.

9. UI STRATEGY & VISUAL REPRESENTATION NEEDS
==============================================================================
To make the deep technical complexity of Machine Learning, Thermodynamics, and 
Blockchain accessible to regular users, investors, and policymakers, the UI 
must serve as an interactive "Digital Twin" of the entire process.

A. Representing Machine Learning (WasteVision AI)
   - Need: Users need to trust that the AI isn't just randomly guessing.
   - UI Strategy: Real-time bounding boxes on a webcam overlay (`WebcamDetector.tsx`). 
     When an item is scanned, the UI must immediately display a "Material Breakdown" 
     (e.g., 80% Organic, 20% Plastic) and show a live calculation of its Methane 
     Potential (kg CH4). It grounds abstract AI in immediate physical reality.

B. Representing Thermodynamics (PlasmaSim)
   - Need: Gasification is an invisible, microscopic process occurring at 5000°C. 
     A simple text output of "Energy Generated" is not convincing.
   - UI Strategy: A step-by-step interactive 3D pipeline (`plasma_reactor_sim.py` 
     mapped to `components/digital-twin/PlasmaCore.tsx`). The UI must visually 
     transport the scanning result into a glowing reactor, showing real-time 
     sensory data (Temperature: 5500°C, Slag dropping, Syngas rising) and 
     gradually filling a "Clean Energy" battery meter. 

C. Representing Blockchain (CCTS SmartMarket)
   - Need: Blockchain ledgers are inherently opaque strings of hex characters.
   - UI Strategy: A "Minting" animation (`LedgerChain.tsx`) that visually converts 
     the virtual syngas into glowing tokens. The marketplace dashboard (`ccts/page.tsx`) 
     should resemble a modern stock exchange, showing live bids from simulated 
     industries (e.g., Airlines buying credits) so the user feels they are participating 
     in a live, breathing economy rather than just looking at a database table.

D. Representing Analytics (Macro Impact)
   - Need: Translating micro-actions (scanning one bottle) to macro-impact (city health).
   - UI Strategy: An interactive city heatmap. As more simulated waste is destroyed in 
     the app, the red "danger zones" around the virtual Ghazipur landfill should 
     visibly shrink and cool down (transition to green), mapped via `analytics/environmental_impact_model.py`.
     This provides a dopamine-driven feedback loop, proving to the user that 
     their actions are healing the environment.
